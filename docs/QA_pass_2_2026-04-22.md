## NEW findings

### MAJOR (5)

1. **Second unbounded realloc path missed by pass-1 fix** — `miner/main/ezshare_client.c:290-295`. `ezshare_file_list_add()` does `realloc(..., new_cap * sizeof(ezshare_file_t))` with `new_cap = capacity * 2` and no upper cap. A malformed or oversized HTML listing with 10K+ entries can exhaust miner heap. Same class of bug as pass-1 #12, separate code path.
2. **Static `cr_hdr` shared across concurrent HTTP requests** — `mule/main/file_server.c:176`. `static char cr_hdr[64]` inside a handler that runs on multiple worker threads; two simultaneous Range requests stomp each other's `Content-Range`. Pass-1 fix moved proxy buffers to stack but missed this one.
3. **65 s mutex hold → `httpd` worker starvation** — `mule/main/file_server.c:90,224-228`. The extended mutex timeout masks 503s but a stuck miner blocks the mutex for 65 s per request. With the default `httpd` worker count (4), a handful of concurrent requests fills all workers; later clients wait behind them and `lwIP` TCP buffers fill while handlers block. Browser directory listings that pipeline 6+ requests trigger this under a slow miner.
4. **`proxy_req` miner state not reset between requests** — `miner/main/scanner_task.c:237-241`. A second `proxy_req` arriving before the first completes overwrites `proxy_req_id` / `proxy_path` in place; the first request's in-flight HTTP fetch then completes under the second's metadata. Mule's mutex normally prevents this, but UART glitches or firmware bugs that duplicate a request do not go through the mutex.
5. **`proxy_chunk` seq never validated on mule** — `mule/main/file_server.c:194-205`. Chunks are reassembled in arrival order without consulting the `seq` field. UART is in-order in normal operation, so this is defense-in-depth, but a late chunk from a previously-timed-out request silently corrupts the response of the next request on the same ID space.

### MEDIUM (7)

6. **`proxy_meta` processable multiple times** — `mule/main/file_server.c:165-191`. Unlike `proxy_chunk`, no req_id match on meta — a stale or duplicated meta overwrites the HTTP status mid-stream.
7. **NVS write race on captive-portal double-submit** — `mule/main/captive_portal.c:203-215`. `set_wifi()` and `set_ezshare()` are two separate NVS commits; two concurrent form submissions can interleave so SSID comes from user A and password from user B. Dev declined in pass 1; a one-line mutex around `handle_save()` would close it.
8. **UART polling starves handler during throughput** — `mule/main/uart_handler.c:147`. `uart_read_bytes(..., 100 ms)` in a tight chunk loop makes the handler uninterruptible during a long transfer. Combined with the 65 s mutex hold and WDT=10 s, headroom is thinner than it looks.
9. **Per-chunk timeout defeatable by 1-byte keepalives** — `mule/main/file_server.c:113-123`. Pass 1 flagged this; still no total timeout cage. A miner trickling one byte per <60 s holds the proxy indefinitely.
10. **`Range` end not validated against file size** — `mule/main/file_server.c:37-60`. `Range: bytes=0-999999999` is forwarded raw; client receives whatever ezShare returns (often 200 + full file) instead of a 416.
11. **Event-callback `data_len` not pre-checked before realloc** — `miner/main/ezshare_client.c:33-44`. The 256 KB cap trips after computing `new_capacity`; correct but fragile if the allocator semantics ever change. Worth a comment.
12. **Captive portal: no confirmed fallback if saved WiFi fails to connect** — `mule/main/wifi_manager.c` + `main.c` boot flow. If the user saves a wrong password, behavior on connect-fail needs confirmation: auto-fallback to portal (good), retry loop forever (bad), or lockout until factory reset (bad).

### NICE TO HAVE (3)

13. **`url_decode` doesn't bounds-check `src[i+1]` / `src[i+2]`** — `mule/main/captive_portal.c:131-146`. Trailing `%A` reads one past the terminator. Reachable only from captive-portal WPA2 peers, nuisance-level.
14. **Base64 malloc per chunk** — `miner/main/scanner_task.c:92-119`. Hundreds of malloc/free on a large file; a static chunk-sized staging buffer removes heap churn.
15. **Mule forwards `dir_param` raw to miner** — `mule/main/file_server.c:254-265`. ezShare's own path scoping is the safety net; a `DATALOG/` prefix canonicalization would be defense-in-depth (noted, not required under LAN-trusted stance).

## Suggested fix order

1. **#1 (file-list realloc cap)** — same pattern as pass-1 #12, trivial cap (e.g. 4096 entries), prevents miner OOM from a malformed directory.
2. **#2 (static `cr_hdr`)** — one-line change to stack-local; real regression risk with browser multi-range probes.
3. **#3 (worker starvation)** — needs a stuck-miner detector that releases the mutex early, or a supervisor task that resets UART after N consecutive timeouts. Harder, but the most user-visible failure.
4. **#4 + #5 (req_id validation)** — pair the same req_id check across all `proxy_*` messages, not just chunks.
