# QA Pass 1 — 2026-04-22

Static code review of `mule/main/` (~1,120 lines) and `miner/main/` (~980 lines). Project is marked **EXPERIMENTAL v0.2**, but the unauthenticated LAN HTTP server and path-forwarding behavior make several findings real production blockers even for a hobby build. Findings are from reading source only — none dynamically verified on hardware.

## MAJOR (8) — crashes, data exposure, stuck states

1. **Proxy mutex held across full UART roundtrip** — `mule/main/file_server.c:90`. If miner crashes mid-transfer, mutex stays held, every subsequent HTTP request returns 503 until reboot.
2. **Path traversal / unauth LAN exposure** — `mule/main/file_server.c:248,264`. `dir=` and `file=` params forwarded raw to ezShare; HTTP server has no auth, anyone on home WiFi can read arbitrary SD paths.
3. **DNS parse stack overflow** — `mule/main/captive_portal.c:56`. `pos += buf[pos] + 1` with no bounds check against `len`; malformed DNS packet can push `pos` past the buffer before the write.
4. **strncpy without NUL term on 511-byte path** — `miner/main/scanner_task.c:235`. `proxy_path` used in later `snprintf` without guaranteed `\0`.
5. **AP SSID not NUL-terminated** — `mule/main/captive_portal.c:254`. 32-byte `strncpy` into `ap_config.ap.ssid` leaves no terminator if input is exactly 32 bytes; WiFi stack expects C strings.
6. **Silent query-param truncation** — `mule/main/file_server.c:237-240,256-258`. `httpd_query_key_value` truncates to 256 bytes with no return-code check, so long paths silently become the wrong path.
7. **Base64 decode output unchecked** — `mule/main/file_server.c:187-189`. `decode_buf[4096]` receives decoded data without validating the written output length before `httpd_resp_send_chunk`.
8. **JSON parse failure loops without buffer clear** — `mule/main/file_server.c:126-127`. Malformed UART line keeps `continue`-ing; can spin until watchdog or starve other tasks.

## MEDIUM (13) — leaks, races, logic errors

9. **Shared static buffers across concurrent HTTP requests** — `mule/main/file_server.c:109-110`. Two clients → corrupted responses.
10. **`s_req_id++` not atomic** — `mule/main/file_server.c:96`. Concurrent requests can share an ID; wrong response delivered to wrong client.
11. **Race on `last_proxy_time_us`** — `miner/main/scanner_task.c:127-133,280`. Read by one task, written by another, no barrier; wrong idle-disconnect decisions.
12. **Unbounded realloc on ezShare response** — `miner/main/ezshare_client.c:37`. Capacity doubles forever; malicious or broken SD content → OOM.
13. **Per-chunk UART timeout 60 s, no overall cap** — `mule/main/file_server.c:113`. Stuck miner sending keepalive-sized chunks never trips the timeout.
14. **`esp_restart()` before UART flush** — `miner/main/scanner_task.c:198`. Factory-reset ACK may never reach the mule.
15. **NVS existence check with uninitialized stack buffer** — `mule/main/nvs_config.c:63,73`. `read_str` can return error without setting `t[4]`; behavior depends on stack state.
16. **cJSON NULL-root not re-checked after parse** — `miner/main/scanner_task.c:219-247`. `cJSON_Delete` is NULL-safe but the intervening `GetObjectItem` calls aren't.
17. **No rate limit on UART parse failures** — `mule/main/file_server.c:126-127`. Flood → watchdog.
18. **HTTP client init failure leaves response struct tracked** — `miner/main/ezshare_client.c:139-140`. Caller may use a freed buffer.
19. **`mbedtls_base64_encode` on zero-length input** — `miner/main/scanner_task.c:94`. `malloc(1)` then encode; edge case with undefined behavior in some mbedTLS versions.
20. **WiFi SSID/password `strncpy` — no NUL guarantee** — `mule/main/wifi_manager.c:71-72`. Same class as #5 but on station config.
21. **URL decode `strtol` accepts invalid hex** — `mule/main/captive_portal.c:131-134`. `%ZZ` silently becomes 0; portal credentials mangled instead of rejected.
22. **`parse_field` uses `strstr` prefix match** — `mule/main/captive_portal.c:142-159`. `user` matches `user_id`; also O(n²).
23. **`uart_send_json` returns success on send failure** — `mule/main/file_server.c:74-79`, `miner/main/scanner_task.c:71-73`. Silent protocol desync.

## NICE TO HAVE (4)

24. **Silent fallback to default ezShare creds** — `miner/main/scanner_task.c:49-60`. If NVS is corrupt, device uses hardcoded SSID/password with no warning log.
25. **`Range:` header parses only single range** — `mule/main/file_server.c:37-60`. Multipart byte ranges ignored; some clients break.
26. **Architecture drift / undocumented protocol** — README describes base64 + JSON-over-UART, code matches, but no documented max file/chunk size or protocol-framing spec. Would bite the next maintainer.
27. **Idle disconnect race not documented** — miner drops ezShare WiFi after 5 min; if a new `proxy_req` arrives exactly at the timeout boundary, first chunk may fail.

---

## Suggested fix order

1. **#2 (path traversal + no auth)** — LAN is not trusted; at minimum validate paths start with `A:DATALOG` and add a shared-secret header.
2. **#1 (mutex deadlock)** — add a stale-lock detector or release the mutex with a timeout so miner crashes don't brick the mule.
3. **#3 (DNS stack overflow)** — trivial to fix (bounds check `pos` every iteration), trivial to exploit on an AP.

## Comparison to cpapdash-push-c3

- No HTTPS/cert-validation risk (this project is LAN-only, no cloud push).
- Fewer leaks, smaller codebase, simpler protocol.
- **More dangerous on a shared home network** due to unauthenticated HTTP server + raw path forwarding.
- Same class of bugs in captive portal DNS parser and `strncpy`-without-NUL patterns — these look like shared code lineage that should be fixed in both.
