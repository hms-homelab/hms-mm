## NEW findings

### MAJOR (2)

1. **`uart_send_json` has no mutex — boot-time race vs HTTP handlers** — `mule/main/uart_handler.c:77-112`. `mule_task` sends the ezShare config at boot +5 s; if an HTTP proxy request arrives in the same window, the two `uart_write_bytes` calls can interleave JSON payloads on the UART line. The miner's parse-failure counter increments, and after 10 consecutive failures the connection drops. User sees "Protocol error" during the first few seconds of operation. Fix: wrap `uart_send_json` in a mutex.
2. **`httpd_start` failure silently skips handler registration** — `mule/main/main.c:70-73`. `if (httpd_start(&server, &http_cfg) == ESP_OK) { ... register handlers ... }` — if `httpd_start` fails, the branch is skipped and `mule_task` starts anyway. Device accepts WiFi but serves no endpoints. No `ESP_LOGE`, no reboot, no diagnostic. Silent degraded mode. Fix: log + reboot or enter a clearly-broken state on startup failure.

### MEDIUM (3)

3. **`proxy_chunk` seq field not validated** — `mule/main/file_server.c:197-243`. Pass-2 fix #5 added an `id` check but not a `seq` order check. Out-of-order or duplicated chunks from a miner-side bug will be forwarded to the HTTP client silently. Defense-in-depth via `expected_seq` tracking would close the gap.
4. **HTTP status passed through without range check** — `mule/main/file_server.c:173`. `int http_status = st_j ? st_j->valueint : 200` with no clamp. Values like `0`, `-1`, `100`, `600` are silently passed through. HTTP clients generally tolerate this, but it can mislead clients about error/success.
5. **Content-Length vs chunk total not reconciled** — `mule/main/file_server.c:165-243`. If miner advertises `cl=1000` in proxy_meta but streams only 500 bytes before `is_last=true`, the client receives a truncated response with no way to detect truncation (the declared `Content-Length` matches what was promised, not what was sent). Fix: track total bytes sent and either flag or abort on mismatch.

### NICE TO HAVE (1)

6. **No cJSON nesting depth limit** — `mule/main/file_server.c:127`, `miner/main/scanner_task.c:222`. cJSON's default recursion limit is ~200; a compromised miner sending `{"a":{"a":{...}}}` could stack-exhaust. Low priority under the LAN-trusted stance but worth configuring explicitly if cJSON exposes a knob.
